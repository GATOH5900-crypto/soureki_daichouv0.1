# 蒼暦台帳 拡張設計要件書 — 索引・事典機能と総合創作補助への道筋

作成日: 2026-07-06 ／ 対象: index.html（schemaVersion 2）からの発展

---

## 0. この文書の位置づけ

蒼暦台帳を「創作世界の年表アプリ」から「創作世界の総合台帳（ワールドバイブル）」へ育てるための設計要件をまとめる。中心となる新機能は次の3つ。

1. **事典（エントリ）**: 人物・地名・組織・用語などを1件ずつ登録する新しいデータ種別
2. **索引**: 事典項目を五十音・種別・タグから引ける一覧画面
3. **語のリンクとオーバーレイ表示**: 年表・伝承のメモ中に登場する語をタップすると、画面遷移せずその場に項目の概要が浮かぶ

あわせて、将来の創作補助機能（関係図・登場年表・執筆メモ等）まで見据えた拡張ロードマップと、既存設計との整合ルールを定める。

---

## 1. 守り続ける設計原則（現行版から継承）

拡張のすべては、以下の原則を壊さない範囲で行う。

| 原則 | 内容 | 拡張時の含意 |
|---|---|---|
| 単一ファイル | index.html 1つで完結。ビルド工程・外部CDN・ライブラリ依存なし | 新機能もすべて素のJS/CSSで書く。ファイル内のセクション構成（状態／保存／検証／描画／道具箱）を維持する |
| オフライン完結 | 端末内 localStorage 保存＋JSON書き出し／読み込み | 事典データも同じJSONに同居させる。サーバ・同期機構は導入しない |
| iPhone最適化 | ホーム画面追加のPWA、セーフエリア対応、44px以上のタップ領域 | オーバーレイ・索引もタッチ前提で設計。ホバー依存のUIを作らない |
| データの可搬性 | JSONは人間が読める形式。未知フィールドは温存する「通り抜け規則」 | schemaVersion 3 へ上げつつ、v2ファイルの読み込み互換を保証する |
| 和のトーン | 明朝体見出し・緋のアクセント・落ち着いた配色 | 新画面も既存のデザイントークン（CSS変数）だけで組む |

**単一ファイルの限界について**: 現在約1,700行。本要件のPhase 2まで実装すると3,500〜4,500行程度と見込む。この規模までは単一ファイルを維持する価値が上回ると判断する。それを超える拡張（関係図エディタ等）が現実になった時点で、分割（ES modules＋結合スクリプト）を別途検討する。

## 2. データモデル拡張（schemaVersion 3）

### 2.1 新データ種別: entries（事典項目）

```jsonc
{
  "app": "soreki-daicho",
  "schemaVersion": 3,
  "meta": { "title": "", "exportedAt": "" },
  "tags":    [ /* 既存のまま */ ],
  "events":  [ /* 既存のまま */ ],
  "legends": [ /* 既存のまま */ ],
  "entries": [
    {
      "id": "ent_xxxx",              // 必須。genEntryId() で生成
      "kind": "person",              // 必須。person/place/org/term/item/other
      "name": "ミシェーラ",           // 必須。正式名（表示名）
      "reading": "みしぇーら",        // 任意。五十音索引の並び順に使う。空なら name で代用
      "aliases": ["ミシェーラ将軍", "北の鉄壁"],  // 任意。自動リンクの検出語にも使う
      "summary": "北方戦線を立て直した大公国の将軍。", // 任意。オーバーレイに出す一行紹介
      "memo": "詳細本文。ここにも [[リンク]] を書ける。",
      "tags": ["tag_04"],            // 既存タグを共用
      "born":  { "year": 1889, "month": null, "day": null },  // 任意（人物向け）
      "died":  null,                 // 任意。null=存命または不明
      "fields": [                    // 任意。自由なキー・値の補足欄
        { "k": "所属", "v": "大公国軍" },
        { "k": "身長", "v": "172cm" }
      ],
      "relations": [                 // 任意。他エントリへの関係
        { "target": "ent_yyyy", "type": "師", "note": "士官学校時代" }
      ],
      "noAutoLink": false,           // true にすると自動リンク検出の対象から外す
      "createdAt": 0, "updatedAt": 0
    }
  ]
}
```

設計判断のポイント:

- **kind は固定6種から始める**（人物・地名・組織・用語・物品・その他）。ユーザー定義種別は将来の候補に留める。索引のフィルタと絵記号表示が単純になる。
- **reading（読み）は任意**。日本語名の五十音ソートは機械的に解決できないため、ユーザーが必要に応じて振る。未入力なら name の NFKC 正規化でソートし「読み未設定」群として扱う。
- **born/died は events と同じ日付構造**（年必須・月日は精度として省略可）を使い回す。既存の `fmtSoreki`・`cmpEvents` 系の道具がそのまま効く。
- **relations は片方向で保存し、表示時に両方向を合成**する（A→Bを保存したら、Bの詳細画面にも「Aから: 師」と出す）。双方向保存は整合性維持のコストが高い。
- **fields は key-value の自由欄**。世界観ごとに必要な属性（種族・爵位・領地…）が違うため、スキーマで縛らない。

### 2.2 検証・正規化・移行

既存の `validateData` / `normalizeData` を拡張する。

- **v2ファイルの読み込み**: `entries` が無ければ `[]` を補う。それ以外は現行の正規化のまま。**v2→v3は無損失・自動**で、ユーザーへの確認は不要（読み込み成功トーストに「v2形式から引き継ぎました」程度の告知は可）。
- **v3ファイルをv2アプリで読む場合**: 現行v2の検証は未知キーを許容し、mergeData も未知キーを温存するため、`entries` はそのまま素通りして保全される（編集はできないが消えない）。この後方互換性は今後も維持する。
- **entries の検証項目**: id/name/kind の型、born/died の日付整合（died < born はエラーではなく警告に留める — 転生・時間遡行等の創作事情を許容）、relations.target の型。**参照先が存在しない relations はエラーにせず「リンク切れ」として表示側で扱う**（部分的なJSONの統合を妨げないため）。
- **タグ削除時のカスケード**に entries を追加。**エントリ削除時**は、(a) 他エントリの relations から該当 target を除去し、(b) 本文中の `[[id:...]]` 記法はリンク切れ表示に落とす（本文の書き換えはしない）。

### 2.3 容量の見通し

localStorage の実用上限を約5MBとすると、1エントリ平均500バイトでも1万件規模まで収まる。個人の創作用途では十分。ただし**保存失敗（QuotaExceeded）を現在は握りつぶしている**ため、拡張時に「自動保存に失敗しています。書き出しでバックアップしてください」のバナー表示を追加する（既存 #banner の仕組みを流用）。

## 3. リンク機構（本設計の核）

メモ本文中の語と事典項目を結びつける方法は2系統を併用する。

### 3.1 明示リンク記法 `[[…]]`

- `[[ミシェーラ]]` — 名前で参照。name または aliases に一致するエントリへ結ぶ
- `[[ent_xxxx|将軍]]` — idで参照し「将軍」と表示。**改名に強い**ため、編集UIから挿入する場合はこちらを生成する
- 一致するエントリが無い場合は「未作成リンク」として点線下線で表示し、タップで**その名前を初期値にした新規エントリ作成シート**を開く（執筆しながら項目を増やす動線。これが創作補助としての要）

### 3.2 自動リンク（辞書マッチ）

- 全エントリの name＋aliases を検出辞書とし、メモ本文の表示時に一致箇所をリンク化する
- 照合は既存の `normText`（NFKC＋小文字化）を通した上での**最長一致優先**。「ミシェーラ将軍」と「ミシェーラ」が両方登録されていれば長い方に結ぶ
- **誤爆対策**:
  - 2文字未満の語は自動リンク対象外（設定で変更可）
  - エントリごとの `noAutoLink` フラグ（「北」「王」のような一般語を名前に持つ項目を守る）
  - 台帳タブに「自動リンク: 入／切」の全体設定を置く（既定は入）
- **実装方式**: 検出語を長い順に並べた交替型の正規表現を1本組み、データ変更時（refresh時）にだけ再構築してキャッシュする。数百エントリ×表示中数十カード程度なら十分速い。Aho-Corasick等の導入はエントリ数千件で遅延が体感されてからでよい
- **描画の安全性**: 現行の `card-memo` は textContent 描画で XSS 安全。リンク化後も **innerHTML は使わず**、テキストを分割して `h()` でテキストノードとリンクボタンを並べる方式とする

### 3.3 逆引き（バックリンク）

リンクの真価は逆方向にある。エントリ詳細画面に以下を自動集計して表示する。

- **登場イベント**: 本文・タイトルにこのエントリへのリンク（明示・自動とも）を含む events を年代順に列挙 → 実質「この人物の年表」
- **登場伝承**: 同様に legends
- **言及エントリ**: 他エントリの本文・relations からの被参照

集計は表示のたびに全文走査すると重いので、**データ変更時に逆引き索引（entryId → 出現先idリスト）を一度だけ構築してメモリに保持**する（保存はしない。導出データはJSONに含めない方針）。

## 4. オーバーレイ表示（年表上での語の参照）

ユーザー体験の中心。**「読んでいる場所を離れずに、語の素性を確かめて、すぐ戻れる」**ことを最優先に設計する。

### 4.1 挙動

1. 年表・伝承・近傍ビューのメモ内リンク語をタップ
2. 画面下から**小型のボトムシート（ピークシート）**が浮かぶ。背後の年表は見えたまま、スクロール位置も保持
3. シートの内容（コンパクト表示）:
   - 種別絵記号＋名前＋読み
   - 生没年（人物なら「蒼暦1889年〜　現在○歳」の自動計算。閲覧中の年を基準にできれば尚良い）
   - summary（一行紹介）
   - タグ、relations の先頭数件
   - アクション: **「全文を見る」**（索引タブの詳細へ遷移）／**「登場箇所を辿る」**（このエントリでフィルタした年表に切替）
4. シート内のリンク（relations や summary 中の語）をさらにタップすると、**シートの中身が差し替わり、ヘッダに「◀ 戻る」が出る**（スタック方式、深さ上限5）
5. 閉じる: 背景タップ／下スワイプ／×ボタン。閉じれば元の年表がそのままの位置で残っている

### 4.2 実装方針

- 既存のレイヤ機構（`sheet-layer` / `showLayer`）を流用し、**新レイヤ `peek-layer`** を1枚追加する。z-index は sheet より下、通常コンテンツより上
- 高さは内容量に応じて最大40vh程度。全文は出さない（全文は索引タブの仕事）
- 履歴スタックは `state.ui.peek = { stack: [entryId, ...] }` で管理。タブ切替時は破棄
- 近傍ビュー（全画面レイヤ）の上でも動くよう z-index を近傍より上に置く

### 4.3 段階的な提供

オーバーレイは自動リンクが無くても価値があるため、**Phase 1 では明示記法 `[[…]]` のリンクだけでオーバーレイまで通す**。自動リンクは Phase 2 で重ねる。

## 5. 索引タブ（事典）

### 5.1 タブ構成の変更

現行の3タブ（年表・伝承・台帳）に**「事典」を加えた4タブ**とする。並びは「年表・伝承・事典・台帳」。FABは事典タブでは「新規エントリ登録」に切り替わる（現行の年表／伝承の切替と同じ流儀）。

### 5.2 一覧（索引）画面

- **並び**: 読み（無ければ名前）の五十音順。「あ・か・さ…」の行見出しを sticky 表示（年表の年見出しと同じ作法）。読み未設定・非かな始まりは末尾「他」にまとめる
- **絞り込み**: 既存の filter バーを共用拡張し、(a) 検索語（name/aliases/summary/memo を対象）、(b) 種別チップ（人物／地名／組織／用語／物品／他）、(c) タグチップ
- **一覧カード**: 種別絵記号・名前・読み・summary 1行・タグ点。タップで展開（年表カードと同じ開閉挙動）
- **五十音ジャンプ**: 画面端の「あかさたな…」インデックスバー、または年ジャンプ相当のダイアログ。エントリ100件超で真価が出る

### 5.3 詳細表示

一覧カードの展開では足りない情報量になるため、エントリ詳細は**全画面レイヤ**（近傍ビューと同じ形式）とする。

- 上段: 名前・読み・種別・タグ・生没年・summary
- 中段: 本文（リンク化適用）、自由フィールド表
- 下段: **関係**（relations 双方向合成）、**登場イベント／伝承**（逆引き。タップでその年表カードへジャンプ — 既存 `jumpToCard` を流用）
- アクション: 編集／このエントリの登場年表を見る／テキスト書き出し（近傍ビューの書き出しと同じ流儀で、人物紹介1件分をコピーできる）

### 5.4 登録・編集シート

イベント登録シートと同じボトムシートで統一する。項目: 名前（必須）／読み／種別（セグメント選択）／別名（カンマ区切り入力）／一行紹介／生没年（人物時のみ表示、期間スイッチと同じ部品を流用）／タグ（既存ピッカー流用）／自由フィールド（行追加式）／関係（エントリ検索→種類ラベル入力）／本文。

**名前を変更した場合**、本文中の `[[旧名]]` 記法が切れる可能性がある。保存時に旧名での明示リンクが何件あるかを数え、「○件のリンクを新しい名前に書き換えますか／別名として旧名を残しますか」を確認ダイアログで提示する（既定は「別名に残す」— 破壊的でない方を既定にする）。

## 6. 年表側の統合強化

- **フィルタの拡張**: 年表・伝承の絞り込みに「エントリで絞る」を追加（タグチップと並べてエントリ検索欄）。オーバーレイの「登場箇所を辿る」はこれを内部的に呼ぶ
- **生没年の投影（任意設定）**: 人物エントリの born/died を年表に薄く表示する（年見出しの「進行中」チップと同じ枠で「存命: ミシェーラ(29)」等）。既定は切。イベントを汚さない
- **近傍ビュー**にもリンク化とオーバーレイを適用する
- **テキスト書き出しの拡張**: 近傍書き出しに「登場エントリ一覧を付ける」オプション。執筆時にAIや他ツールへ文脈ごと渡す用途を想定

## 7. 実装フェーズ計画

| Phase | 内容 | 規模感 |
|---|---|---|
| **1. 事典の骨格** | schemaVersion 3・entries CRUD・事典タブ（索引一覧＋詳細＋登録シート）・明示リンク `[[…]]` のパースとオーバーレイ（ピークシート）・未作成リンクからの新規作成動線 | 中。既存部品の流用が効く |
| **2. リンクの知能化** | 自動リンク検出（辞書マッチ＋noAutoLink＋全体設定）・逆引き索引と「登場イベント」表示・エントリフィルタ・名前変更時のリンク追従ダイアログ | 中 |
| **3. 創作補助の深化** | 生没年の年表投影と年齢計算・関係の簡易図示（SVG手描きの放射図。閲覧専用から始める）・エントリ単位のテキスト書き出し拡充・統計への事典項目追加 | 大。個別に価値判断して着手 |
| **将来候補（要件のみ記録）** | 複数台帳（作品）の切替、章・シーンの執筆メモ管理、地図画像へのピン打ち、系図表示、ユーザー定義の種別 | — |

各Phaseは**単独でリリース可能**な単位とする。Phase 1完了時点で「索引＋オーバーレイ」というユーザーの当初要望は満たされる。

## 8. 非機能要件

### 8.1 性能

- 描画は現行同様「タブ全体の再構築」を基本とするが、**カード本文（メモのリンク化を含む）は開いた時に初めて生成する遅延化**を導入する。リンク化は文字列走査を伴うため、閉じたカードでは行わない
- 自動リンク辞書（正規表現）と逆引き索引は**データ変更時のみ再構築**。目標: エントリ500件・イベント1,000件でタブ切替1秒未満（iPhone実機基準）
- 上記を超える規模で問題が出た場合の逃げ道（仮想スクロール、増分描画）は設計に織り込まず、実測してから対処する

### 8.2 互換性・データ保全

- v2 JSONの読み込み無損失保証（§2.2）
- 統合（merge）読み込みは entries にも同じ「id一致は上書き・他は追加」規則を適用
- 既存のバックアップ世代機構（BACKUP_KEY）はそのまま entries を含む全体に効く
- 保存失敗の可視化（§2.3）

### 8.3 操作性・アクセシビリティ

- 本文中のリンクは `<button>` で実装し、下線＋わずかな色差で示す（緋アクセントは強すぎるので、リンク専用の控えめな色をトークンに追加: 例 `--link:#7d94b5` 系）
- タップ領域はリンク語が小さくても padding で44px相当を確保しつつ、行間を崩さない（負マージン相殺）
- オーバーレイは `prefers-reduced-motion` 尊重（既存の全停止規則が効く）
- 検索・照合は一貫して `normText`（NFKC）を通し、全半角・かな大小の揺れを吸収

### 8.4 書き心地（創作補助としての質）

- 記法は `[[…]]` の1種類だけに留め、Markdown等は導入しない（「メモは平文・改行はそのまま保存」が現行の思想）
- 「書いている途中で項目を作れる」（未作成リンク→新規作成）ことを最優先動線にする
- エントリ登録の必須項目は**名前だけ**。まず名前だけ放り込み、後から育てる使い方を妨げない

## 9. 主なリスクと採った立場

| リスク | 立場 |
|---|---|
| 自動リンクの誤爆（一般語と同名の項目） | 2文字未満除外＋noAutoLinkフラグ＋全体オフ設定の三段構え。完璧は狙わず、明示記法を確実な逃げ道として残す |
| 同名の別項目（同姓の親子等） | 自動リンクは「最初に一致した1件」ではなく、複数一致時はオーバーレイに候補一覧を出して選ばせる |
| 名前変更でリンク切れ | id記法の推奨（編集UIが生成するのはid記法）＋変更時の追従ダイアログ（§5.4） |
| 単一ファイルの肥大 | Phase 2完了時点で約4,500行までは許容。超えたら分割を別途設計 |
| localStorage消失（iOSの7日規則等） | 現行と同じく「JSONの書き出しが正」の思想を貫き、保存失敗の可視化と書き出し導線の強化で対処。同期機能は作らない |

## 10. 画面イメージ（文字ワイヤー）

